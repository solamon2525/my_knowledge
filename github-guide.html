คู่มือการใช้งาน GitHub ฉบับสมบูรณ์

ยินดีต้อนรับเข้าสู่โลกของ Version Control คู่มือนี้จะเปลี่ยนคุณจากผู้เริ่มต้นให้กลายเป็นนักพัฒนาที่ใช้งาน GitHub ได้อย่างมืออาชีพ ไม่ว่าจะทำงานคนเดียวหรือเป็นทีม

สิ่งที่คุณจะได้เรียนรู้

  • ความแตกต่างระหว่าง Git และ GitHub
  • การสร้าง Repository และจัดการไฟล์
  • การทำงานด้วย Branch และ Pull Request
  • GitHub Pages, Actions และฟีเจอร์ขั้นสูง
  • Workflow มาตรฐานในการทำงานเป็นทีม
Level 1 — Basic

ตอนที่ 1: พื้นฐานและการเตรียมตัว

1.1 Git และ GitHub คืออะไร?

Git: เปรียบเสมือน "เครื่องย้อนเวลา" ของโค้ด ทำหน้าที่บันทึกทุกความเปลี่ยนแปลงในไฟล์ รันบนเครื่องของคุณโดยไม่ต้องเชื่อมอินเทอร์เน็ต

GitHub: คือ "บ้าน" ของ Git ที่อยู่บนอินเทอร์เน็ต ช่วยให้เราแชร์งาน สำรองข้อมูล และทำงานร่วมกับคนอื่นได้ทั่วโลก

1.2 การตั้งค่าเริ่มต้น (ทำครั้งเดียว)

# บอก Git ว่าคุณคือใคร
git config --global user.name "ณัฐพงศ์ สิงห์ชมภู"
git config --global user.email "your@email.com"

# ตรวจสอบการตั้งค่า
git config --list

1.3 คำสั่งพื้นฐาน 5 ตัวที่ใช้บ่อยที่สุด

git init          # เริ่มใช้ Git ในโฟลเดอร์ปัจจุบัน
git add .         # เตรียมไฟล์ทั้งหมดเพื่อ commit
git commit -m ""  # บันทึก snapshot ของโค้ด
git push          # ส่งขึ้น GitHub
git pull          # ดึงล่าสุดจาก GitHub

1.4 ขั้นตอน Clone และ Push ครั้งแรก

  1. สร้าง Repository บน GitHub.com (กด New Repository)
  2. git clone https://github.com/username/repo.git — โคลนลงเครื่อง
  3. สร้างหรือแก้ไขไฟล์ในโฟลเดอร์นั้น
  4. git add . → git commit -m "ข้อความ"
  5. git push origin main — ส่งขึ้น GitHub

Mission 1: First Commit

สร้าง Repository แรกบน GitHub แล้ว Clone ลงมาในเครื่อง จากนั้นสร้างไฟล์ hello.txt พิมพ์ข้อความว่า "สวัสดี GitHub!" แล้ว Push กลับขึ้นไปดูซิ!

Level 2 — Intermediate

ตอนที่ 2: การทำงานร่วมกันและ Branching

การทำงานจริงเราไม่ได้ทำคนเดียว การใช้ Branch จะช่วยให้เราลองฟีเจอร์ใหม่ๆ ได้โดยไม่ทำโค้ดหลักพัง

2.1 การทำงานกับ Branch

git branch                     # ดู branch ทั้งหมด
git branch feature-new-ui      # สร้าง branch ใหม่
git checkout feature-new-ui    # สลับไปที่ branch นั้น
git checkout -b feature-login  # สร้างและสลับในคำสั่งเดียว
git merge feature-new-ui       # รวม branch กลับมา main
git branch -d feature-new-ui   # ลบ branch ทิ้งหลัง merge

2.2 Pull Request (PR) คืออะไร?

คือการส่ง "คำขอ" ให้เจ้าของโปรเจกต์ตรวจสอบโค้ดของเราก่อนที่จะรวม (Merge) เข้าไปจริงๆ ทีมมืออาชีพทุกทีมใช้ PR เป็นมาตรฐาน

ขั้นตอนวิธีทำ
1. Fork repo ต้นทางกดปุ่ม Fork บน GitHub
2. Clone มาที่เครื่องgit clone URL
3. สร้าง branch ใหม่git checkout -b my-fix
4. แก้ไขโค้ด + commitgit add . && git commit
5. Push branch ขึ้นไปgit push origin my-fix
6. เปิด Pull Requestกดปุ่ม "Compare & pull request"

2.3 แก้ไข Merge Conflict

เกิดเมื่อสองคนแก้ไขไฟล์เดียวกัน บรรทัดเดียวกัน Git จะแสดงเครื่องหมายพิเศษ:

<<<<<<< HEAD
โค้ดของฉัน (branch ปัจจุบัน)
=======
โค้ดของเพื่อน (branch ที่กำลัง merge)
>>>>>>> feature-branch

วิธีแก้: เปิดไฟล์ เลือกเอาส่วนที่ต้องการ ลบเครื่องหมาย แล้ว commit ใหม่

Level 3 — Advanced

ตอนที่ 3: ฟีเจอร์ขั้นสูง

3.1 GitHub Pages — โฮสต์เว็บฟรี

คุณสามารถโฮสต์เว็บไซต์ HTML ของคุณได้ฟรีๆ ที่ username.github.io/repo-name

  1. Push ไฟล์ index.html ขึ้น GitHub
  2. ไปที่ Settings → Pages
  3. เลือก Source: Deploy from branch → main
  4. รอ 1-2 นาที เว็บจะ live อัตโนมัติ

3.2 GitHub Actions — CI/CD อัตโนมัติ

ระบบ Automation ที่ช่วยรันงานอัตโนมัติทุกครั้งที่มีการ Push เช่น รัน test, build, deploy

# .github/workflows/deploy.yml
name: Deploy to Pages
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Deploy
        run: echo "Deploying..."

3.3 .gitignore — ซ่อนไฟล์ที่ไม่ต้องการ

# ตัวอย่าง .gitignore
node_modules/
.env
*.log
.DS_Store
dist/

ลองนำไฟล์ HTML อัปโหลดขึ้น GitHub แล้วเปิดใช้งาน GitHub Pages เพื่อให้คนทั่วโลกเข้าชมผ่าน URL ของคุณเอง! เช่น solamon2525.github.io/my_knowledge

การบริหารโปรเจกต์บน GitHub

Issues — ติดตามปัญหาและงาน

ใช้สร้าง to-do list ของโปรเจกต์ รายงานบั๊ก ขอฟีเจอร์ใหม่ หรือติดตามงานในทีม

Projects — Kanban Board

จัดการงานด้วย board แบบ Trello ได้เลยในหน้า GitHub ตั้ง column: Todo → In Progress → Done

Wiki — เอกสารโปรเจกต์

พื้นที่เขียนเอกสารประกอบโปรเจกต์ เหมาะสำหรับ user manual หรือ API documentation

ฟีเจอร์ใช้ทำอะไรเหมาะกับ
Issuesติดตามบั๊กและงานทุกโปรเจกต์
ProjectsKanban boardทีมขนาดกลาง-ใหญ่
Wikiเอกสาร/คู่มือโปรเจกต์ open-source
Discussionsพูดคุย/ถามตอบชุมชนนักพัฒนา

GitHub Flow — มาตรฐานการทำงานเป็นทีม

  1. สร้าง branch จาก main สำหรับทุกฟีเจอร์/บั๊กฟิกซ์
  2. commit บ่อยๆ พร้อมข้อความที่อ่านเข้าใจ
  3. Push branch ขึ้น GitHub และเปิด Pull Request
  4. รีวิวโค้ด กับทีม แก้ไขตามคำแนะนำ
  5. Merge เมื่อผ่านการรีวิว และลบ branch ทิ้ง
  6. Deploy จาก main เสมอ — main คือ production

Commit Message ที่ดี

✅ feat: เพิ่มระบบ login ด้วย Google
✅ fix: แก้บั๊กปุ่ม submit กดแล้วไม่ทำงาน
✅ docs: อัปเดต README วิธีติดตั้ง
❌ "แก้แล้ว"  ← ไม่รู้ว่าแก้อะไร
❌ "update"   ← ไม่มีประโยชน์
